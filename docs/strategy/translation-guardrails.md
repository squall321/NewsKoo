# Translation & Rewriting Guardrails

NewsKoo는 비용 효율적인 AI 파이프라인과 글로벌 콘텐츠 저작권·약관을 모두 준수하기 위해 아래 지침을 따른다.

## 비용 제약 및 하드웨어 가정
- 번역 및 생성 모델은 **RTX 5070 Ti 16GB VRAM** 환경에서 실행 가능해야 한다.
- Hugging Face, GitHub 등에서 제공되는 **오픈소스 모델**을 우선 활용하고, 필요 시 LoRA/QLoRA 등 경량화 기법을 적용한다.
- 상용 번역 API(예: DeepL, Google) 호출은 예외 케이스로 제한하며, 호출 비용이 발생하는 경우 페이즈 계획에 명시하고 사전 승인 후 사용한다.
- 모든 학습·추론 워크로드는 가급적 스케줄링하여 GPU 유휴 시간을 줄이고, CPU 전용 백오프 경로도 마련한다.

## 콘텐츠 재구성 원칙
- Reddit 등 소스 사이트의 약관을 준수하기 위해 **직역 대신 요약·패러프레이즈** 방식으로 핵심 유머만 추출한다.
- 자동 파이프라인은 핵심 사건/농담/밈을 구조화된 형태로 추출하고, **서비스 고유의 스토리텔링 템플릿**을 적용해 다시 작성한다.
- 검수자는 최종 게시 전에 톤앤매너(브랜드 가이드 참조)와 문화적 민감성을 재검토한다.
- 원문 링크, 작성자 표기, 면책 문구 등을 메타데이터로 관리하여 투명성을 확보한다.

## 번역 파이프라인 단계
1. **크롤링**: 허용된 사이트만 수집하고 Robots.txt 및 API Rate Limit를 준수한다.
2. **요약/정규화**: 원문을 문단·대화 등 구조로 분해하고, 중복 또는 민감 정보를 제거한다.
3. **로컬 번역**: Hugging Face에서 제공하는 7B~13B 범위의 번역/지시형 모델을 QLoRA 혹은 4bit 양자화하여 사용한다.
4. **스타일 재작성**: 사전 정의된 "NewsKoo 스타일 가이드"를 프롬프트로 적용하여 콘텐츠를 재구성한다.
5. **인간 검수**: 어색한 표현, 문화적 문제, 저작권 위반 가능성을 확인하고, 필요 시 직접 수정한다.
6. **기록 보관**: 사용된 모델 버전, 프롬프트, 수동 수정 내역을 `docs/records/phase-XXX/`에 보관해 재현성을 확보한다.

## 추후 API 통합 전략
- 상용 번역 API를 추가할 경우에도 **자동 재작성 모듈**을 통해 원문을 그대로 노출하지 않는다.
- API 호출 비용이 월 예산을 초과하면 자동으로 로컬 모델 경로로 폴백하도록 설계한다.
- 커뮤니티 피드백이나 신규 소스가 추가되면 우선 로컬 모델로 실험하고, 성능이 부족한 경우에만 API 기반 옵션을 켠다.

이 가이드는 `docs/phases/`의 모든 페이즈에서 공통적으로 준수해야 하며, 변경 사항이 생길 경우 해당 페이즈 문서와 함께 개정 이력을 남긴다.
